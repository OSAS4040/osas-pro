# سجل التطوير والتحديث — جلسة 16 أبريل 2026

> **الغرض:** وثيقة مرجعية واحدة تلتقط حوكمة التفعيل، خارطة الطريق، تجربة المستخدم والألوان، وربط الصلاحيات، مع **منع التداخل** بين عمل الفريق و**ترتيب الملفات**. أي تغيير لاحق يُفضّل الإشارة إلى هذا الملف في وصف الـ PR.

---

## 1) خطاب مهني قصير (للإدارة والعملاء الداخليين)

نُدار منصة متعددة المستأجرين مع مسارات مالية حساسة؛ لذلك لا نُقيّم «التفعيل» بالظهور في القائمة فقط، بل بمعيار واضح: **API سليم + واجهة متزامنة + صلاحية أو باب ميزة + دليل قبول موقّع أو اختبار تلقائي**.  
مسارات **إدارة المنصة** (`/api/v1/platform/*`) منفصلة سياسياً عن مسارات **المستأجر**؛ لا تُطبَّق عليها نفس افتراضات نطاق الشركة إلا حيث صُمّمت صراحة.  
أي توسعة (إضافات، ذكاء، امتثال) تمرّ أولاً بـ**ورقة نطاق صفحة واحدة** (معرّف، أولوية، معيار قبول، مخاطر) قبل البرمجة — ذلك يقلّل التداخل بين الفرق ويحافظ على جودة الإصدار.

### سياسة التفعيل: مرحلي — مع الاختبارات والأمان والاحتراف

**ليس هدفنا «تفعيل كل المزايا في كل الأقسام دفعة واحدة».** الهدف هو **تفعيل مرحلي ومسيطر عليه**: موجة تلو الأخرى، بحيث يخرج كل إصدار بحدود واضحة وقابل للمراجعة.

| الركيزة | المقصود العملي |
|--------|----------------|
| **مرحلي** | كل موجة تغلق نطاقاً محدداً (مثلاً: تفعيل موحّد للقوائم، ثم تشغيلي، ثم HR…) مع معالم تسليم؛ لا تُفتح كل البوابات دون جرد. |
| **اختبارات** | لكل ميزة تُفعَّل رسمياً: اختبار تلقائي (Feature/Unit حيث ينطبق) **أو** قائمة قبول يدوية موقّعة؛ الانحدار يُراقب قبل وبعد الدمج. |
| **أمان** | المسارات المالية والمنصّة تحت مراجعة صلاحيات + حدود معدّل + تتبع `trace_id`؛ لا تفعيل يتجاوز `tenant` أو `platform.admin` بدون تصميم. |
| **احتراف** | رسائل خطأ واضحة، وثائق قصيرة في الـ PR، وعدم إظهار للمستخدم ما ليس مكتملاً أو غير ممنوح له (قائمة + API متسقان). |

هذه السياسة **متوافقة** مع أقسام الوثيقة التالية (المراحل 0–7): هي إطار زمني وتنظيمي للتفعيل **التدريجي**، لا لقائمة «تشغيل الكل».

## 2) تعريف «مفعّل» (Definition of Done — تشغيلي)

| المعيار | الوصف |
|--------|--------|
| API | مسار HTTP يعيد 2xx لسيناريو سعيد، و403/422 بمعنى واضح (باقة مقابل دور مقابل تعطيل منتج). |
| واجهة | الشاشة تقرأ وتكتب الحقول نفسها التي يتوقعها الـ API دون تعارض. |
| صلاحية / ميزة | مفتاح في `config/permissions.php` أو `platform.permission:*` أو `TenantBusinessFeatures` / `system/capabilities` موثّق. |
| قبول | اختبار Feature/Unit **أو** قائمة تحقق يدوية **موقّعة** من المالك المنتجي. |

**غير مفعّل رسمياً:** placeholder في الواجهة بدون API، أو بند قائمة يفتح شاشة فارغة — يُعالَج إما بإخفاء حتى الجاهزية أو بربط `feature_gate` حقيقي.

## 3) الفصل: منصة مقابل مستأجر

| النطاق | مسارات تقريبية | IAM |
|--------|----------------|-----|
| المستأجر | `/api/v1/*` تحت `tenant` + اشتراك + صلاحيات `permission:*` | أدوار الشركة في `config/permissions.php` |
| المنصة | `/api/v1/platform/*` | `EnsurePlatformAdmin` + `platform.permission:*` في `config/platform_permissions.php` و `config/platform_roles.php` |

لا تُمزج صلاحيات المنصة مع صلاحيات المستأجر في واجهة واحدة دون تصميم صريح.

## 4) المراحل 0–7 (ملخّص تنفيذي)

### المرحلة 0 — جرد وقياس (≈ 1–2 أسبوع)

| النشاط | المخرجات |
|--------|----------|
| كتالوج من `GET /api/v1/system/capabilities` + `TenantBusinessFeatures` + `config/system_capabilities.php` | مصفوفة: ميزة → باب → باقة → دور |
| مطابقة كل بند في الشريط الجانبي مع route + API | [`docs/Tenant_Navigation_API_Map.md`](./Tenant_Navigation_API_Map.md) — قائمة → API → 403/placeholder |
| تحديث `docs/Platform_Module_Status_Matrix.md` بعد الجرد | لقطة FULL / PARTIAL |
| 10–20 سيناريو قبول (POS، أمر عمل، فاتورة، قيد، حضور، ترقية باقة، …) | معيار «جاهز للإنتاج» |
| اختبارات Feature لمسارات حديثة (شحن محفظة + ملف نشاط) | `WalletTopUpTransferInstructionsTest`، تأكيد `fixed_assets` في `ProductionReadinessCompanyFeatureProfileTest` |

### المرحلة 1 — طبقة التفعيل الموحّدة (≈ 2–3 أسابيع)

- ربط شاشات «ترقية / معطّل» بـ `feature_gate` أو إخفاء القائمة حتى اكتمال المنتج.  
- مراجعة `config/permissions.php` و `routes/api.php` لكل وحدة HR / إداري / محاسب.  
- توثيق `PlanAddon` / `SubscriptionAddonEntitlements` واختبار «إضافات الاشتراك» من الواجهة إلى الـ API.  
- توحيد رسائل 403: **باقة** مقابل **دور** مقابل **تعطيل إداري**.

### المرحلة 2 — التشغيل التجاري (≈ 3–6 أسابيع)

إغلاق PARTIAL (عروض، CRM، اجتماعات، إلغاءات أوامر عمل)، المشتريات/الموردون، وتقليل ازدواجية مسارات المحفظة تدريجياً.

### المرحلة 3 — الموارد البشرية (ورشة) (≈ 3–5 أسابيع)

اختبارات Feature لمسارات `workshop/*`؛ «اتصالات إدارية» إما API أو إخفاء؛ عدم بيع «HR مؤسسي» قبل وجوده في المصفوفة.

### المرحلة 4 — المالية والمحاسبة (≈ 4–8 أسابيع)

انحدار للـ FULL؛ مسار فرعي لـ ZATCA/GL؛ قرار منتج لـ Wave 7 (إقفال، خزينة، …).

### المرحلة 5 — الإداري والتكاملات (≈ 3–5 أسابيع)

كل بند = قائمة تحقق API + سياسة وصول؛ أمن API keys والويبهوكس.

### المرحلة 6 — التحليلات والاشتراك والإضافات (موازية)

BI / مركز العمليات ببيانات حقيقية؛ تعريف عقد كل إضافة (واجهة+خلفية أم تسويق فقط)؛ إطلاق تدريجي.

### المرحلة 7 — التصلب والتشغيل (مستمر)

PHPUnit/Pest، `trace_id`، طوابير، مراجعة تغييرات مالية.

## 5) تجربة المستخدم والألوان (مبدأي — دون تغيير عشوائي للهوية)

| المبدأ | التطبيق في المستودع |
|--------|---------------------|
| مصدر الألوان | إعدادات الشركة: الثيم في `SettingsView` + `ui_theme` المحفوظ للشركة. |
| RTL أولاً | الواجهات العربية: `dir="rtl"` في الصفحات القياسية (`app-shell-page`). |
| تباين مقروء | تجنّب رمادي فاتح على أبيض لنصوص طويلة؛ استخدام `dark:` للوضع الداكن. |
| لا «ألوان زائدة» في الـ PDF الرسمي | مستندات مثل تعليمات التحويل: ألوان هادئة (نيلي/تركواز خفيف) وليست إعلاناً بصرياً. |
| رسالة واضحة عند عدم الجاهزية | 422 مع نص يوجّه لإعداد الحسابات (مثال: شحن المحفظة بدون حسابات بنكية). |

أي تغيير شامل للهوية يمرّ بمراجعة منتج + لقطة شاشة قبل/بعد في الـ PR.

## 6) الصلاحيات والأدوار — عدم إظهار إلا ما يُمنح

### الملفات المرجعية

| الملف | الدور |
|-------|--------|
| `backend/config/permissions.php` | أدوار المستأجر + قائمة المفاتيح لكل دور |
| `backend/config/platform_permissions.php` | مفاتيح صلاحيات المنصة |
| `backend/config/platform_roles.php` | ربط أدوار مشغّلي المنصة بالمفاتيح |
| `frontend/src/layouts/AppLayout.vue` | تصفية عناصر القائمة عبر `auth.hasPermission` و `requiresPermission` |
| `frontend/src/views/about/CapabilitiesView.vue` (إن وُجد) + `GET /system/capabilities` | كتالوج آمن للمستخدم |

### نمط مطلوب لكل صفحة جديدة

1. مفتاح صلاحية فريد في `permissions.php` (إن لم يوجد).  
2. إدراج المفتاح في الأدوار المناسبة (`owner` / `manager` / …) أو عبر واجهة إدارة الأدوار إن وُجدت.  
3. في `AppLayout.vue` (أو وحدة التنقل المركزية): `v-if="auth.hasPermission('…')"` أو إدخال في مصفوفة العناصر مع `requiresPermission`.  
4. في `api.php`: `middleware('permission:…')` أو `platform.permission:…`.  
5. اختبار: مستخدم بدون الصلاحية لا يرى الرابط **ولا** يستطيع استدعاء الـ API (403).

### منصة vs موظف

- **مشغّل المنصة:** `EnsurePlatformAdmin` + `SaasPlatformAccess` — لا يُعرض له شريط المستأجر إلا إذا كان حساباً مختلطاً بتصميم واضح.  
- **موظف المستأجر:** يُرشّح عبر `system/capabilities` + الصلاحيات؛ أي عنصر لا يملك له مفتاحاً يُخفى.

## 8) ما نحتاجه لاحقاً لجدولة أسبوعية مرقّمة

1. قائمة الإضافات (معرّفات + أولوية).  
2. أفق زمني (8 أسابيع / ربع سنة) وبيئة (تجريبي / إنتاج).  
3. ترتيب القطاعات (تشغيلي / مالي / HR أولاً).

## 9) الخلاصة

هذا الملف يجمع **الحوكمة** و**خطة الموجات** و**تجربة المستخدم** و**استراتيجية الصلاحيات** في مكان واحد، ويربط التعديلات البرمجية الأخيرة بمساراتها لتقليل التداخل. **التفعيل مرحلي دائماً** — مع الاختبارات والأمان والاحتراف كركائز — وليس إطلاق كل الأقسام دفعة واحدة. التزام الفريق بمعيار «مفعّل» في القسم 2 هو الضمانة لأن **المستخدم لا يرى إلا ما يملك صلاحية وميزة له**، مع استمرار تحسين التصفية في الشريط الجانبي حسب القسم 6.
