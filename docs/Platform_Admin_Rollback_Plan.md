# خطة الرجوع (Rollback) — إدارة المنصة

## متى نقرر Rollback

- فشل التحقق بعد النشر في مسارات منصة حرجة.
- خلل صلاحيات يسبب تسريب وصول.
- فشل مستمر في API إدارة المنصة بعد محاولات إصلاح سريعة.

## من يقرر Rollback

- قائد الإطلاق (Release Owner) بالاتفاق مع:
  - مسؤول التشغيل (Ops)
  - ممثل المنصة (Platform Owner)

## خطوات Rollback

1. إعلان `NO-GO` رسمي وإيقاف أي تفعيل إضافي.
2. إعادة التطبيق إلى الإصدار السابق المستقر (artifact/tag السابق).
3. إعادة تشغيل الخدمات المرتبطة وفق إجراء النشر القياسي.
4. التحقق من المسارات الأساسية في الإصدار السابق.
5. توثيق السبب والوقت والأثر.

## ما الذي لا يجوز أثناء Rollback

- لا rollback جزئي غير موثق.
- لا تعديلات عشوائية مباشرة على قاعدة البيانات.
- لا تجاوز لصلاحيات أو تعطيل حراس الأمان.

## التحقق بعد Rollback

- تسجيل دخول مشغّل المنصة يعمل.
- `/platform/overview` تعمل.
- لا أخطاء API حرجة على `/api/v1/platform/*`.
- نتيجة Smoke مختصرة = PASS.

## في حال فشل rollback الجزئي

1. تثبيت الحالة (freeze deployment).
2. استدعاء مسؤول قاعدة البيانات/المنصة فورًا.
3. تنفيذ rollback كامل وفق النسخة الاحتياطية/الإصدار السابق.
4. رفع Incident تشغيلي ومتابعة حتى الاستقرار.
