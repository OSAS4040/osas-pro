# أساس التوسّع: جوال، وسائط، وأداء

**الغرض:** توحيد الاتجاه التقني لبدء تطبيق جوال (مثلاً Flutter) ولرفع جودة معالجة الصور والبيانات دون تكرار منطق الأعمال.

## 1) واجهة واحدة للحقيقة (API)

- الاستمرار على **`/api/v1`** كعقد ثابت للويب الحالي وأي عميل جوال.
- المصادقة للجوال: توكنات (Sanctum Personal Access Token أو مسار يعادلها) — تصميم صريح بعيداً عن اعتماد الكوكيز وحدها.

## 2) تخزين الملفات القابل للتوسع

- المتغير **`MEDIA_TENANT_UPLOAD_DISK`** (قيمة `public` أو `s3`) يضبط أين تُحفظ رفعات المستأجر: الشعارات، صور الفواتير، مستندات المركبات، العقود، مرفقات المشتريات، إلخ.
- عند التحويل إلى **`s3`**: اضبط **`AWS_*`** في `.env` وتأكد من صلاحيات الحاوية والـ CORS إن لزم.
- مسارات المجلدات تبقى منظمة تحت بادئات مثل `companies/{id}/…` و`invoices/{id}/…` لتسهيل الأرشفة والسياسات لاحقاً.

## 3) صور أخف وأسرع من المتصفح

- **`frontend/src/utils/imagePipeline.ts`**: تصغير أطول ضلع وإعادة ترميز JPEG قبل إرسال base64 للـ API (مثلاً من **`PlateScanner`**).
- ضبط اختياري عبر **`VITE_IMAGE_UPLOAD_MAX_EDGE`** و **`VITE_IMAGE_UPLOAD_JPEG_QUALITY`** في واجهة البناء.

## 4) خطوات لاحقة (مقترحة بالترتيب)

1. **PWA** على نفس Vue لتثبيت خفيف للعميل/الأسطول.
2. **Laravel Reverb** (أو طبقة WebSocket) لتنبيهات حالة أوامر العمل/الدفع — بعد تثبيت متطلبات Docker.
3. **تطبيق Flutter** يركّز على Staff (أوامر عمل) مع **مزامنة تدريجية**؛ تجنّب «أوفلاين كامل» في الإصدار الأول.
4. **طوابير** للعمليات الثقيلة (زاتكا، توليد PDF/XML) كما في التوصيات السابقة — لا حظر واجهة المستخدم على ردود الجهات الخارجية.
