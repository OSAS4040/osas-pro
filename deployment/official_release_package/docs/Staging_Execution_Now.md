## الترتيب الآمن الكامل (إلزامي — لا تتخطَوْا خطوة)

اتبعوا هذا التسلسل **من 1 إلى 8**. الانتقال لخطوة لاحقة قبل إتمام السابقة يزيد خطر نشر إعدادات خطأ أو تجاوز صلاحيات.

| # | ماذا | أين | ملاحظة أمنية |
|---|------|-----|----------------|
| 1 | **دمج عبر PR** إلى `main` فقط؛ فحص **Policy env on PR** ناجح | GitHub | لا أسرار في الكود؛ راجع [`Branch_Protection_And_Review_Policy.md`](./Branch_Protection_And_Review_Policy.md) |
| 2 | **`policy-env-example`** | محلياً: `node scripts/check-policy-env-example.mjs` أو `make policy-env-example` | يتحقق أن أمثلة الـ env لا تُفعّل `SAAS_ALLOW_TENANT_PLAN_CATALOG_EDIT=true` |
| 3 | **`staging-gate`** | محلياً مع Docker: `make staging-gate` | Vitest + PHPUnit مسار المنصة/SaaS؛ انظر [`Makefile`](../Makefile) |
| 4 | **(اختياري موصى به قبل إصدار كبير)** **`verify`** | `make verify` | lint + build واجهة + **كل** `php artisan test` |
| 5 | **رفع Staging** | خادمكم | أسرار من [`backend/.env.staging.example`](../backend/.env.staging.example) و[`frontend/env.staging.example`](../frontend/env.staging.example) — **لا ترفعوا `.env` إلى git**؛ `APP_DEBUG=false`؛ `SAAS_ALLOW_TENANT_PLAN_CATALOG_EDIT=false`؛ بريد منصة **ضيق** في `SAAS_PLATFORM_ADMIN_EMAILS` |
| 6 | **الاختبار اليدوي** على **عنوان Staging الحقيقي** | متصفح + API | الجدول في القسم التالي + [`Staging_Manual_Test_Checklist.md`](./Staging_Manual_Test_Checklist.md) |
| 7 | **تسجيل النتيجة** | تذكرة/تقرير | نموذج التسجيل في أسفل هذا الملف؛ قرار صريح: PASS / PASS مع ملاحظات / FAIL |
| 8 | **الإنتاج** | فقط بعد **PASS** (أو ملاحظات غير حرجة حسب سياسة الفريق) | [`Staging_Deploy_Runbook.md`](./Staging_Deploy_Runbook.md) — **نفس القيود**؛ لا توسّعوا الصلاحيات «للتجربة» في الإنتاج |

**قاعدة:** لا تنشروا على **الإنتاج** قبل إكمال **5–7** بنجاح وفق السياسة. الفحوص 1–4 لا تغني عن اليدوي على Staging.

## الآن: البنود اليدوية فقط على Staging الحقيقي

بعد أن يكون **Staging** شغّالاً وقد مرّ **policy-env-example** و**staging-gate** آلياً، نفّذوا **فقط** ما يلي على عنوان Staging الفعلي:

| # | البند |
|---|--------|
| 1 | **دخول staff** |
| 2 | **اختبار المسارات الأساسية** (لا شاشة بيضاء، لا 404 غير متوقع) |
| 3 | فتح **`/about/taxonomy`** |
| 4 | **Admin API:** بريد **مصرَّح** → **200** على `GET /api/v1/admin/companies`؛ بريد **غير مصرَّح** → **403** (يفضّل رمز `PLATFORM_OPERATOR_REQUIRED`) |
| 5 | **بوابة معطّلة** (`VITE_ENABLED_PORTALS` في build Staging): التحقق من **redirect** صحيح (لا حلقة، لا صفحة بيضاء) |
| 6 | **مراقبة:** logs، queues، **4xx / 5xx** |

---

## متى نعتبر Staging **PASS كاملاً**؟

- **PASS آلي** (policy + staging-gate) **و**
- **PASS يدوي:** البنود أعلاه **كلها** ناجحة **بدون** مشاكل **حرجة**.

عندها تكون النتيجة: **PASS كامل للـ Staging** — والقرار المنطقي التالي: **نقل نفس الإعدادات المحافظة إلى الإنتاج** (انظر [`Staging_Deploy_Runbook.md`](./Staging_Deploy_Runbook.md)).

---

## متى نوقف / لا ننتقل للإنتاج؟

إذا ظهر في **الاختبار اليدوي** أي مما يلي (أو ما يعادله حرجاً):

- خطأ **صلاحيات** غير متوقع
- **route** مكسور
- **صفحة بيضاء**
- **5xx** متكرر أو على مسار حرج
- **redirect** غير صحيح للبوابات المعطّلة

فالحالة:

| الوضع | التسمية |
|--------|---------|
| الآلي ناجح واليدوي فيه ما سبق | **PASS آلي** + **FAIL** أو **PASS مع ملاحظات حرجة** في Staging اليدوي — **لا انتقال** للإنتاج حتى يُعالَج ويُعاد الاختبار |

## بالترتيب فقط (ملخص)

الترتيب التفصيلي والأمني في الجدول **«الترتيب الآمن الكامل»** أعلاه. هنا ملخص قصير:

1. **ارفعوا Staging** (خادم + واجهة + أسرار بيئة محافظة — انظر [`Staging_Deploy_Runbook.md`](./Staging_Deploy_Runbook.md)).
2. شغّلوا:
   - `make policy-env-example` أو `node scripts/check-policy-env-example.mjs`
   - `make staging-gate` (يتطلب Docker حسب المشروع)
3. (اختياري) `make verify` قبل دمج/إصدار كبير.
4. **اتبعوا** [`Staging_Manual_Test_Checklist.md`](./Staging_Manual_Test_Checklist.md) **بالكامل**.
5. **سجّلوا نتيجة كل بند:** `PASS` أو `FAIL` مع **ملاحظة مختصرة** عند الفشل.
6. **راقبوا:** logs، queues، 4xx/5xx (انظر قسم «هـ» في القائمة اليدوية).

---

## قرار الانتقال للإنتاج

| الحالة | إجراء |
|--------|--------|
| مرت القائمة **بالكامل** بـ PASS | يمكن الانتقال لـ **نفس الإعدادات المحافظة** في الإنتاج |
| **PASS مع ملاحظات** وطيفة و**غير حرجة** | يمكن الانتقال مع توثيق الملاحظات ومتابعتها |
| **FAIL** حقيقي، أو ملاحظات **حرجة** | **يتوقف** الانتقال — يُعالَج السبب — ثم **تُعاد** نفس القائمة من البداية |

---

## ما لا نفعله الآن

- لا نبدأ **ميزات جديدة** خارج إغلاق Staging.
- لا نفتح **اشتراك عميل ↔ منصة**.
- لا نبني **محفظة منصة** كاملة / دفتر إيرادات منصة منفصل.
- لا نوسّع **Admin/Platform** بلا حاجة تشغيلية معتمدة.
- لا نغيّر **نموذج البيانات** أو مسارات مالية جديدة قبل إغلاق الحلقة.

---

## المخرج الواحد المطلوب بعد الاختبار

أهم شيء الآن هو **مخرج واحد واضح** — نتيجة Staging اليدوي:

| الحالة | المعنى |
|--------|--------|
| **PASS** | كل البنود الحرجة ناجحة |
| **PASS مع ملاحظات** | نجاح عام مع نقاط غير حرجة موثّقة |
| **FAIL** | وجود فشل حرج يمنع الإنتاج حتى يُعالَج |
