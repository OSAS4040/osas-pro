# بوابات التنفيذ — خطوة كاملة ثم اختبار ثم التالي

**المبدأ:** لا تُعتبر أي خطوة في خارطة الطريق **مكتملة** حتى تكون **صحيحة وموثّقة**، و**تمرّ الاختبارات المرتبطة بها** (آليًا أو يدويًا حسب تعريف الخطوة). **لا تبدأ الخطوة التالية** قبل إغلاق الخطوة الحالية.

## 1) تعريف «إغلاق الخطوة»

لكل خطوة، يجب تحقيق **كل** ما يلي:

| البند | المعنى |
|--------|--------|
| **نطاق واضح** | ما الذي يدخل الخطوة وما الذي يُستبعد صراحةً. |
| **دمج/نشر محلي** | الكود في المستودع؛ البناء (frontend/backend) ناجٍ في بيئة الفريق. |
| **اختبار مرتبط** | حسب الجدول أدناه — لا يُكمل «تم» بدون نتيجة نجاح أو تقرير سبب التخطي المعتمد. |
| **وثيقة** | تحديث README/رودماب/ADR عندما تفرض الخطوة ذلك. |

## 2) حد أدنى للاختبار حسب نوع التغيير

| نوع التغيير | قبل الانتقال للخطوة التالية |
|-------------|------------------------------|
| **واجهة (Vue)** | `npm run type-check` و `npm run test` (Vitest) و `npm run build` من `frontend/`. |
| **خلفية (Laravel)** | تشغيل PHPUnit للمجلدات أو `--filter` ذات الصلة. **داخل Docker:** إن فشل `php artisan test` باتصال `127.0.0.1:5432`، استخدم **`./vendor/bin/phpunit`** من مجلد `backend` داخل الحاوية `app` — يحمّل `phpunit.xml` (مثل `DB_HOST=postgres`) بشكل موثوق. |
| **تخزين / رفع ملفات** | اختبار يدوي أو Feature لمسار رفع واحد + التحقق من الحذف عند الاستبدال. |
| **نشر Staging** | `Staging_Manual_Test_Checklist.md` + أي `staging-gate` المعتمد في المشروع. |

## 3) ترتيب العمل اليومي (مقترح)

1. اختر **خطوة واحدة** من `Roadmap_Four_Weeks_Staff_First.md` (أو وثيقة المشروع الرسمية).
2. نفّذ التغيير ضمن النطاق فقط.
3. شغّل **فحوصات البوابة** للنوع (الجدول أعلاه).
4. إن فشل فحص: **أصلح** أو **ألغِ** جزء النطاق — لا تضيف ميزة جديدة على نفس الفرع قبل الإصلاح.
5. سجّل في PR/التذكرة: أوامر التشغيل والنتيجة (مثلاً: `Tests: 45 passed`).
6. انتقل للخطوة التالية.

## 4) استكشاف أخطاء سريع

| العرض | إجراء |
|--------|--------|
| `Class "Spatie\Permission\PermissionServiceProvider" not found` | كاش حزم قديم بعد إزالة حزمة. داخل حاوية `app`: احذف `bootstrap/cache/packages.php` و`services.php` ثم `php artisan package:discover`. |
| اتصال PostgreSQL بـ `127.0.0.1` أثناء الاختبار في Docker | استخدم `./vendor/bin/phpunit` (انظر جدول الخلفية أعلاه) أو اضبط `DB_HOST=postgres` في `.env` المستخدم مع `php artisan test`. |

## 5) استثناءات (تحتاج موافقة صريحة)

- اختبار معطّل مؤقتًا: يُفتح تذكرة **P0/P1** مع سبب وتاريخ إزالة العطل.
- بيئة بدون Docker/DB: لا تُعلَن الخطوة «مكتملة إنتاجيًا» — تُسمّى «جاهز للمراجعة فقط».
