# خطة 4 أسابيع — أولوية فريق الورشة (Staff)

**انضباط التنفيذ:** اتبع [`Execution_Gates_Sequential_Workflow.md`](./Execution_Gates_Sequential_Workflow.md) — **أكمل كل أسبوع/خطوة بالاختبارات المطلوبة قبل البدء بالتالية.**

**الافتراض:** المستخدم الأول الذي نُسعِدُه خلال أربعة أسابيع هو **من يعمل يومياً داخل التشغيل**: استقبال مركبة، أمر عمل، تحديث حالة، ربط فاتورة/دفع عند الحاجة — وليس بالضرورة «مدير أسطول» أو «محاسب فقط».

**ما لن نبنيه في نفس الأربعة أسابيع (حد واضح):** تطبيق Flutter كامل، أوفلاين كامل مع مزامنة، ذكاء تعلّم آلي من الصفر، تكامل زاتكا إنتاجي جديد إن لم يكن جاهزاً أصلاً في بيئتكم.

---

## الأسبوع 1 — تثبيت ومسار تجريبي

| الهدف | إجراء ملموس في المشروع |
|--------|-------------------------|
| بيئة موثوقة | اتباع **`Staging_Execution_Now.md`** + **`Staging_Deploy_Runbook.md`**: بناء الواجهة، nginx، هجرات، Redis/طوابير. |
| تخزين الملفات | التحقق من **`MEDIA_TENANT_UPLOAD_DISK`** (محلي أولاً) و`storage:link` عند الحاجة؛ جاهزية **S3** كخيار لاحق دون تغيير كود الرفع. |
| مسار حرج واحد | اختبار يدوي موثّق: **تسجيل دخول Staff → قائمة أوامر العمل → فتح أمر → تحديث حالة** (`/work-orders`, `/work-orders/:id`). |
| عقد API خفيف | مسار أو مساران من **`/api/v1`** يُراقَبان (زمن + كود 200) — قاعدة لاحقاً لـ contract tests من OpenAPI. |

**مخرجات:** Staging يعمل؛ قائمة تحقق يدوية موقّعة؛ لا ميزات منتج جديدة إن لم تكن البيئة مستقرة.

---

## الأسبوع 2 — تقليل الاحتكاك اليومي للورشة

| الهدف | إجراء ملموس |
|--------|-------------|
| سرعة الوصول | اختصار من **`AppLayout` / البحث السريع** (إن وُجد) نحو **أوامر العمل** و**مركبة جديدة**؛ توحيد مصطلحات RTL. |
| كاميرا / لوحة | الاعتماد على **`PlateScanner`** + **`imagePipeline`** (موجودان)؛ تدريب فريق مختصر على الإضاءة والزاوية. |
| «ورشة مزدحمة» (MVP) | **منفّذ في المستودع:** متجر Pinia `staffUi` + زر **«وضع سريع»** في **`AppLayout`** عند مسارات `/work-orders`؛ يبسّط **`WorkOrderListView`** و **`WorkOrderShowView`**؛ الاختيار يُحفظ في `localStorage`؛ اختياري **`VITE_STAFF_COMPACT_UI=true`** للتفعيل الافتراضي عند البناء. |

**مخرجات:** فيديو أو لقطات «قبل/بعد» لمسار أمر عمل ≤ 2 دقيقة تحت الضغط؛ راجع **`docs/adr/0001-staff-compact-ui-mode.md`**.

## الأسبوع 3 — قناة واتساب كتشغيل (حلقة مغلقة واحدة)

| الهدف | إجراء ملموس |
|--------|-------------|
| إعداد | مراجعة **`IntegrationsView`** (تبويب واتساب) وربط مزوّد حقيقي أو بيئة اختبار حسب عقدكم. |
| حالة واحدة فقط | اختيار حدث واحد: مثلاً **«أمر عمل جاهز للتسليم»** أو **«تم إنشاء فاتورة»** → إشعار واتساب للعميل برابط **بوابة العميل** أو رقم مرجعي (بدون بناء بوت كامل). |
| الخادم | إن لم يوجد endpoint جاهز: **Job** في Laravel + قالب رسالة قصير؛ إعادة المحاولة عبر الطابور الحالية (**Redis**). |

**مخرجات:** سيناريو E2E واحد: تغيير حالة في الواجهة → وصول رسالة على رقم تجريبي.

## الأسبوع 4 — جاهزية للجوال / الذكاء دون تضخيم النطاق

| الهدف | إجراء ملموس |
|--------|-------------|
| جوال لاحقاً | مستودع **Flutter** فارغ أو شاشة دخول + استدعاء **`POST /api/v1/...`** لقائمة أوامر عمل (قراءة فقط) — يستفيد من **`TenantUploadDisk`** وعقد API الثابت. |
| أو PWA بدلاً منه | **`vite-plugin-pwa`**: تثبيت من المتصفح + كاش shell للـ Staff على أجهزة الورشة الثابتة. |
| ذكاء تدريجي | زر **«تصحيح OCR»** أو حقل ملاحظة يُحفظ مع مرجع المستند (سياسة خصوصية) — ربط لاحق بتحسين **`OcrController`** / خدمات **`IntelligentReading`**. |
| قرار معماري | أول ملف **`docs/adr/0001-....md`**: مثلاً «لماذا PWA قبل Flutter» أو العكس — ليثبت القرار للفريق. |

**مخرجات:** إما binary Flutter تجريبي أو PWA قابل للتثبيت؛ وملف ADR واحد على الأقل.

## مؤشرات نجاح (بسيطة)

- زمن مسار **«فتح أمر → تحديث حالة»** على Staging تحت حد متفق عليه (مثلاً &lt; 30 ثانية تفاعلياً).
- صفر هجرات معلّقة على Staging بعد كل نشر.
- حلقة **واتساب** واحدة تعمل نهاية لنهاية.
- وثيقة **ADR** + تحديث **`Architecture_Mobile_Media_Scale.md`** إذا تغيّر اتجاه التخزين أو الجوال.

## ماذا بعد الأربعة أسابيع؟

- توسيع **أوفلاين قراءة** (SQLite في Flutter) لمجال ضيق، أو تعميق **PWA**.
- **مدير أسطول** أو **محاسب** كجمهور ثانٍ — خطة موازية لا تدمج كل الأدوار في نفس السبرنت.

*هذا الملف يكمّل الاقتراحات الاستراتيجية السابقة ويحوّلها إلى خطوات قابلة للتنفيذ على هذا المستودع.*
