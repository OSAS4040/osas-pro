# تقييم جاهزية اختبار التحميل — WorkshopOS

هذا المستند **أداة قرار تشغيلية**: حدّث الأرقام بعد كل تعرض لـ `make load-test` من `load-testing/reports/latest-summary.json` و`latest.md`.

## 1. مستوى الجاهزية الحالي (أدوات + هيكل)

| البند | الحالة |
|--------|--------|
| هيكل k6 قابل لإعادة الاستخدام (`k6/lib`, `suite.js`) | **مُنفَّذ** |
| سيناريوهات واقعية (جلسة قراءة، POS + idempotency، عزل مستأجر) | **مُنفَّذ** |
| تقارير تلقائية (`reports/latest.md`, `latest-summary.json`) | **مُنفَّذ** |
| دمج Makefile (`make load-test`) | **مُنفَّذ** |
| فحص مسبق بدون ضغط (`k6/preflight.js`) | **مُنفَّذ** |

**الخلاصة:** البنية التحتية لاختبار التحميل **جاهزة مهنياً**. نجاح التشغيل الفعلي يعتمد على بيئة API مُشغَّلة ومُبذورة بالمستخدمين التجريبيين (انظر `env.example`).

## 2. ما يُتحقق منه أثناء التشغيل

- **الملفات الشخصية:** `smoke` · `normal` · `peak` · `stress` · `spike` · `soak` عبر `K6_PROFILE` (انظر `load-testing/README.md`).
- **عتبات k6:** `k6/config/acceptance.js` — تشمل p50/p95/p99 (حسب الملف الشخصي)، `http_req_failed`، `server_errors_5xx`، `client_timeout_or_network`، عزل المستأجر، وidempotency.
- **تقرير stress:** أول «تدهور» و«انهيار» يُحسبان من `STRESS_REPORT_CRITERIA` في نفس الملف (fail rate / p95 لكل مرحلة).
- **الأداء:** `http_req_duration` (p50/p95/p99 في التقرير).
- **الاستقرار:** نسبة الأخطاء تحت حدود الملف الشخصي.
- **سلامة المسارات الحرجة (على مستوى HTTP):** استجابة POS `201` + قراءة فاتورة لاحقة؛ مفاتيح idempotency فريدة تقلل ازدواجية المبيعات.
- **التزامن:** `ramping-arrival-rate` على POS + قراءات متزامنة من مستخدمين افتراضيين متعددين.
- **عزل الشركات:** استدعاء `GET /v1/companies/{معرف_شركة_الجار}` بتوكن مستأجر آخر — يُتوقع **403 أو 404**.

> **تحديد سلامة أرصدة DB/مخزون عميق** يتجاوز k6 وحده؛ يُكمّل بـ `php artisan test`، أو `simulation:stress`، أو فحوص SQL/تقارير بعد التشغيل.

## 3. الاختناقات الشائعة (مألوفة في Docker المحلي)

1. **PHP-FPM:** عدد workers مقابل طلبات متزامنة.
2. **PostgreSQL:** حجم اتصالات البركة؛ استعلامات غير مفهرسة تحت الحمل.
3. **Redis:** قفل أو ذاكرة عند ضغط الجلسات والطوابير.
4. **شبكة Docker:** `host.docker.internal` وترجمة المنافذ على Linux.
5. **القرص:** I/O عند سجلات مفرطة.

## 4. أخطاء تشغيل متوقعة

| العرض | الإجراء |
|--------|---------|
| `setup` فشل تسجيل الدخول | شغّل `php artisan migrate --seed` أو على الأقل `DemoCompanySeeder` و`DemoOperationsSeeder`؛ تحقق من `K6_BASE_URL`. |
| `401` على `/auth/login` | تأكد من وجود المستخدم في قاعدة البيانات وكلمة المرور كما في البذور. |
| `connection refused` من حاوية k6 | تأكد أن nginx يستمع على المنفذ 80 على المضيف؛ جرّب `K6_BASE_URL=http://host.docker.internal/api`. |
| POS يُستثنى (`pos_skipped`) | مستأجر A بلا عملاء/منتجات — أعد البذور أو استخدم بريداً لمستخدم شركة المحاكاة. |

## 5. الأولويات التحسينية (بعد أول تشغيل ناجح)

1. قياس p95 تحت الحمل الهادف للإنتاج وتثبيت **SLO** (مثلاً p95 < 500ms لقراءات، < 2s للكتابة المالية).
2. ضبط **php-fpm** `pm.max_children` وفق الذاكرة المتاحة.
3. مراقبة **آثار استعلام بطيء** في PostgreSQL.
4. فصل **طابور المعاملات المالية** إن لزم (مع بقاء الاتساق عبر idempotency الحالية).

## 6. هل ننتقل للمرحلة التالية؟

- **نعم (جاهزية هندسية):** تم تسليم **إطار k6**، التقارير، والدمج مع `make verify` سابقاً؛ لا تعارض مع ESLint.
- **نعم (تشغيل إنتاج / staging):** يتطلب **تشغيل ناجح واحد على الأقل** لـ `make load-test` أو ما يعادله، مع مراجعة `latest.md` وعدم تجاوز عتبات الفشل — ثم توثيق الأرقام في هذا الملف.

## 7. ملاحظات البيئة (مُثبتة)

- **Sentry:** المتغيرات `SENTRY_LARAVEL_DSN` / `VITE_SENTRY_DSN` غير مضبوطة افتراضياً → تحذيرات فقط عند `docker compose`.
- **docker-compose `version`:** حقل قديم؛ يُمكن إزالته لاحقاً لتنظيف التحذير.
- **Laravel tests:** مُفترض تشغيلها عبر `docker compose exec app php artisan test` (كما في `make verify`).
